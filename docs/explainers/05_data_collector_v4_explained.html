1. ภาพรวม

DataCollector_v4.mq5 คือ Expert Advisor (EA) ใน MetaTrader 5 ที่ทำหน้าที่ "เก็บข้อมูลกราฟ" ออกมาเป็น CSV เพื่อเอาไป train RL — ไม่ใช่ EA เทรด

MT5 (live / Strategy Tester) ↓ OnTick (ทุกแท่งใหม่) DataCollector_v4 (compute features + label per bar) ↓ FileWrite MQL5/Files/training_data_*.csv ← 67 columns (OHLCV + indicators + candles + target) ↓ Python rl_train.py PPO model (rl_gbpusd.zip)
หัวใจ: EA เปิดบน chart → ทุกแท่งใหม่ คำนวณ indicators & pattern → เขียน 1 row ใน CSV → จบ tester ได้ไฟล์ training พร้อม

2
OnTick — เขียน 1 row ต่อแท่งใหม่ (หัวใจ)
ทุกแท่งใหม่: เช็คเงื่อนไข → ดึง OHLCV → คำนวณ features → เขียน CSV row 
void OnTick() {
  // ⚡ เช็ค new bar (ป้องกัน trigger ทุก tick)
  if(t0[0] == g_lastBarTime) return;
  g_lastBarTime = t0[0];

  // ⚡ ต้องมี history พอ — รวม indicator warmup + window + ForwardBars
  needed = max_period + max_window + ForwardBars + 5;
  if(Bars(_Symbol, _Period) < needed) return;

  // ⭐ จุดสำคัญ: เขียนข้อมูลของบาร์ที่ shift = ForwardBars+1 (อดีต)
  //    เพราะต้องการ future_return = ต้องรู้อนาคต ForwardBars แท่ง
  int i = ForwardBars + 1;

  // ดึง OHLCV ของบาร์นั้น
  o = rates[i].open;  h = rates[i].high;  l = rates[i].low;  c = rates[i].close;

  // สร้าง row
  row = TimeToString(rates[i].time) + "," + _Symbol + ...

  // เพิ่ม features per indicator
  WriteIndicatorValues(RSI_Mode, g_h_rsi, ...)
      // ถ้า AGGREGATE → CalcAggregate(values[]) → 4 columns: min, max, mean, std
      // ถ้า MULTI     → values[0], values[1], ... → N columns

  ทำซ้ำกับ ATR, EMA, Stoch, CCI, WPR, ADX

  // Single-period
  row += macd + "," + macd_signal + "," + macd_hist;
  row += bb_position;

  // Candle patterns 10 ตัว (ถ้าเปิด)
  for(p = 0; p < 10; p++)
      row += CopyBuffer(g_h_candles, p, i, 1);

  // ⭐ Label: future_return = (close[i-ForwardBars] - close[i]) / close[i]
  fr = (closes[i - ForwardBars] - closes[i]) / closes[i];
  row += fr + "," + (fr > +0.001 ? "UP" :
                  fr < -0.001 ? "DOWN" : "FLAT");

  FileWrite(g_handle, row);
}
3
CalcAggregate — สร้าง min/max/mean/std จาก array ของ periods
หัวใจของ FEAT_AGGREGATE mode 
void CalcAggregate(double &values[], double &out_min, &out_max,
                              &out_mean, &out_std) {
  int n = ArraySize(values);
  double sum = 0, sum_sq = 0;
  out_min = values[0]; out_max = values[0];

  for(int i = 0; i < n; i++) {
      if(values[i] < out_min) out_min = values[i];
      if(values[i] > out_max) out_max = values[i];
      sum    += values[i];
      sum_sq += values[i] * values[i];
  }
  out_mean = sum / n;
  out_std  = MathSqrt(sum_sq / n - out_mean * out_mean);
}
จุดต้องสังเกต: n = จำนวน periods ที่ user ตั้ง
ถ้า user ตั้ง RSI_Min=5, RSI_Max=30 → n=26 → mean ของ 26 ค่า
แต่ RL_Indicators.mqh ตั้ง RSI_PMIN=4, RSI_PMAX=30 → n=27 → mean ของ 27 ค่า
→ ค่า rsi_mean ต่างกัน!

OnDeinit — Cleanup ตอนปิด tester
ปิดไฟล์ + ปล่อย indicator handles 
void OnDeinit(const int reason) {
  FileClose(g_handle);

3. Configurable Inputs — ทำไม v4 ถึง flexible (และอันตราย)

v4 เปิดให้ user ปรับ ทุก indicator ได้ผ่าน enum + min/max/step:

enum FeatureMode {
   FEAT_OFF,        // ไม่เก็บ indicator นี้
   FEAT_AGGREGATE,  // ⭐ default — รวม min/max/mean/std (4 columns)
   FEAT_MULTI,      // เก็บแต่ละ period แยกคอลัมน์ (N columns)
};

input FeatureMode RSI_Mode  = FEAT_AGGREGATE;
input int         RSI_Min   = 5;        // เริ่ม period
input int         RSI_Max   = 30;       // จบ period
input int         RSI_Step  = 1;        // step
input string      RSI_Multi = "7,14,20,30";  // list ถ้า MULTI mode

// ทำซ้ำกับ ATR, EMA, Stoch, CCI, WPR, ADX → รวม ~30 inputs

+ Candle patterns toggles (10 ตัว) + thresholds (~12 ตัว) → รวมแล้ว 50+ inputs

ข้อดี: ทดลองง่าย — เปลี่ยน periods แล้วรัน Tester ใหม่ ได้ feature set ใหม่ทันที

4. ทำไม Parity แตก — ต้นตอของ bug rl_gbpusd

ปัญหาคือ design นี้คือ "2 implementations ที่ทำงานคล้ายกัน แต่ไม่ใช่ตัวเดียวกัน":

┌────────────────────────────┐ ┌────────────────────────────┐ │ DataCollector_v4.mq5 │ │ rl_gbpusd_EA.mq5 │ │ (gen training data) │ │ (live inference) │ │ │ │ │ │ • own iRSI/iATR/.. handles │ │ • own iRSI/iATR/.. handles │ │ • own CalcAggregate() │ │ • own CalcAggregate() │ │ • own BuildHeader() │ │ • via RL_Indicators.mqh │ │ • periods = INPUTS (runtime)│ │ • periods = #define (compile)│ └────────────────────────────┘ └────────────────────────────┘ │ │ ▼ ▼ feature values feature values └───────────── ต่างกัน! ─────────────┘ mismatch → bug

กรณีจริงที่พังบน rl_gbpusd:

FeatureCSV (จาก generator เก่า)EA (RL_Indicators)ผล
ema_200 0.017 (distance) 1.756 (raw price) +344σ outlier 💥
cci_min 3.31 -44.0 58% mismatch
adx_std 5.20 4.24 34% mismatch
stoch_std 15.44 14.47 25% mismatch
... รวม 13/58 features เพี้ยน
ผล: feature ที่ส่งเข้า model ตอนรันใน MT5 = out-of-distribution → model output Hold ด้วย confidence 0.97 ตลอด → ไม่เทรด 2003-2007 (เทรดแค่ช่วง warmup ก่อน CSV เริ่ม)

ทำไมถึงเป็นแบบนี้?

  • training_data_v4_GBPUSD.csv อาจถูก gen ด้วย DataCollector รุ่นเก่ากว่า v4 ที่หายไปแล้ว
  • v4 ที่ใช้ตอนนี้เขียน ema_200 เป็น raw price (เหมือน EA) — แต่ CSV เก่ามี distance
  • RL_Indicators.mqh comment บอกชัด: "MUST match DataCollector_v3.mq5" — แต่ v3 ก็เปลี่ยน config ได้
  • ไม่มีการ sync ระหว่าง 2 codebases → drift ไปคนละทาง

5. เทียบกับ DataCollector_RL ใหม่

DataCollector_v4 (เก่า)

  • เสี่ยง own feature code
  • own iRSI/iATR handles
  • own CalcAggregate
  • periods configurable via inputs
  • Header สร้าง dynamic ตาม inputs
  • Future return คำนวณ inline
  • Parity กับ EA: manual sync

DataCollector_RL (ใหม่)

  • ปลอดภัย ใช้ RL_Indicators.mqh
  • handles via RL_InitIndicators()
  • CalcAggregate จาก RL_Indicators
  • periods override ได้ผ่าน inputs
  • Header = RL_ALL_FEATURES[75] คงที่
  • Future return คำนวณใน Python
  • Parity กับ EA: ✅ รับประกัน (shared code)
┌──────────────────────────┐ │ DataCollector_RL.mq5 │ ← input InpRSI_Pmax=30 (default = parity) │ OnInit: RSI_PMAX = 30 │ ← write global ใน .mqh └─────────────┬────────────┘ │ #include ▼ ┌──────────────────────────┐ │ RL_Indicators.mqh │ │ int RSI_PMAX = 30; │ ← global (เคยเป็น #define) │ │ │ RL_BuildAllFeatures() │ ← ใช้ RSI_PMAX └──────────────────────────┘ ▲ │ #include ┌─────────────┴────────────┐ │ rl_gbpusd_EA.mq5 │ ← ไม่ override → ใช้ default = 30 └──────────────────────────┘ ↑ ทั้ง 2 ใช้ .mqh เดียวกัน → feature values ตรงกัน 100%

6. บทเรียนสถาปัตยกรรม

หลักการที่ต้องจำ

Training data + Inference code = ต้องใช้ feature implementation เดียวกัน

  • ถ้าแยก 2 codebase → ไม่ช้าก็เร็วจะ drift → mismatch
  • วิธีปลอดภัยที่สุด = shared code library (.mqh ตัวเดียว) ที่ทั้ง collector + EA include
  • ถ้าต้องการ flexibility → expose เป็น runtime variable (global) ไม่ใช่ #define
  • Default ของ shared code = behavior ของ EA — collector ที่ใช้ default จะให้ data ตรง EA เสมอ

ก่อนแก้: 2 codebase

DataCollector_v4 ─→ CSV (features set A) ─→ Model trained on A ↓ deploy in MT5 ↓ rl_gbpusd_EA ←─ features set B ←─ RL_Indicators.mqh A ≠ B → model เจอ input นอก distribution → output Hold conf 0.97 → ❌ ไม่เทรด

หลังแก้: shared code

┌─ RL_Indicators.mqh ─┐ │ (feature code A) │ └────┬────────┬───────┘ │ │ #include │ │ #include ▼ ▼ DataCollector_RL.mq5 rl_gbpusd_EA.mq5 │ │ ▼ ▼ CSV (A) inference (A) ↓ ↑ Model trained on A ──→ deploy → A == A → ✅ เทรดตามที่เทรนมา
บทสรุป: DataCollector_v4 = stand-alone tool ที่ flexible แต่ต้อง sync manual กับ EA
→ DataCollector_RL = deployment-aligned ที่ใช้ shared code → ผิดไม่ได้
