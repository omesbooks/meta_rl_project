1. UP/DOWN/FLAT คืออะไร?

ทุกแท่งกราฟใน CSV จะถูก label ว่าเป็นหนึ่งใน 3 class ตาม future_return = ผลตอบแทนใน N แท่งข้างหน้า:

UP

future_return ≥ +threshold

เช่น +0.5% ขึ้นไป

"ราคาจะขึ้นแรง"

FLAT

−threshold < future_return < +threshold

เช่น ±0.5%

"ราคาเฉยๆ"

DOWN

future_return ≤ −threshold

เช่น −0.5% หรือลง

"ราคาจะลงแรง"

กฎ threshold = 0.001 (0.1%) คือ default ของ build_training_from_collector.py mode fixed.

ตัวอย่างใน CSV

timestamp           close    future_return  target
2004-01-15 12:00    1.8175   +0.0035        UP      // +0.35% > +0.1%
2004-01-15 16:00    1.8205   -0.0002        FLAT    // |-0.02%| < 0.1%
2004-01-15 20:00    1.8198   -0.0042        DOWN    // -0.42% < -0.1%

2. Class Imbalance — ปัญหาแบบไหน?

เกิดเมื่อ class หนึ่งมีจำนวน เยอะกว่าตัวอื่นมาก → model "ขี้เกียจ" จะ predict class ใหญ่อย่างเดียวก็ได้คะแนนสูง

ตัวอย่าง CSV ที่ imbalanced (ตลาด sideways)

FLAT
14,000 rows
70%
UP
3,600
18%
DOWN
2,400
12%
Lazy classifier trap: ถ้าให้ model classify FLAT ทั้งหมด
→ accuracy = 70% ทันที (เพราะ FLAT มี 70%)
→ ดูเหมือนเก่ง แต่ โง่ (ทายไม่เคยถูกแม้แต่ UP/DOWN ครั้งเดียว)

หลัง relabel (Quantile 33/33/33)

FLAT
6,666
33%
UP
6,667
33%
DOWN
6,667
33%

→ model ต้อง เรียนรู้จริงๆ ทำนายให้ถูกใน 3 class ที่เท่ากัน

3. วิธี Relabel — 3 ตัวเลือก

① Quantile 33/33/33 ⭐ (recommended)

เรียง future_return ทั้งหมด → แบ่งเป็น 3 ส่วนเท่าๆ กัน

# สมมติเรียงค่า future_return ทั้ง 20,000 rows
q1 = quantile(future_return, 1/3)   # เช่น = -0.0008
q2 = quantile(future_return, 2/3)   # เช่น = +0.0009

target = "DOWN"  if  future_return ≤ q1
target = "UP"    if  future_return ≥ q2
target = "FLAT"  otherwise

✅ Balanced เสมอ โดย adaptive threshold ตามการกระจายจริงของ data

② Fixed Threshold

กำหนด threshold ตรงๆ (เช่น ±0.1%)

target = "UP"    if  future_return ≥  +0.001
target = "DOWN"  if  future_return ≤  -0.001
target = "FLAT"  otherwise

⚠️ จำนวน class ขึ้นกับลักษณะตลาด
   ตลาดคึกคัก → UP/DOWN เยอะ, FLAT น้อย
   ตลาด sideways → FLAT เยอะ (imbalance!)

③ Binary UP/DOWN (ไม่มี FLAT)

target = "UP"    if  future_return > 0
target = "DOWN"  if  future_return ≤ 0
# ไม่มี FLAT

ใช้กับ supervised แบบ binary classification เท่านั้น
   ไม่เหมาะกับ workflow ที่มี 3 actions (Buy/Sell/Hold)

4. ทำไมไม่จำเป็นเท่าไหร่ใน RL workflow

กุญแจ: RL ไม่ใช้ target column ในการ train เลย!
มันใช้ reward function ที่คำนวณจาก PnL จริง

หลักฐานใน rl_train.py

# rl_train.py drop target ทิ้งทันทีตอน load:
leaky = [c for c in df.columns
         if any(k in c.lower() for k in (
            "future_", "forward_", "next_", "target"))]
df = df.drop(columns=leaky)
# → "target" หายไปก่อนเข้า PPO

RL เรียนจาก Reward (ใน trading_env.py)

reward = Net-PnL × 50             # ค่าหลัก: คำนวณจาก close prices
       + bonus if pnl > 0.5%   # reward shaping
       - 0.005                  # penalty ตอนเปิด trade
       - 0.0001                 # idle penalty
       ...

# ไม่มี "target" อยู่ในสมการนี้เลย!

เทียบ Supervised vs RL

Supervised (เก่า — PyCaret) RL (workflow ตอนนี้)
Learning signal target column (UP/DOWN/FLAT) reward function (PnL-based)
Class imbalance affects? ✅ มาก — model ขี้เกียจ predict class ใหญ่ ❌ ไม่มี — model ไม่เห็น target เลย
Relabel จำเป็น? ✅ จำเป็น ❌ Skip ได้ 95%
ข้อยกเว้น: build_training_from_collector.py mode quantile (default) ทำให้ target balanced อยู่แล้ว → ไม่ต้อง relabel เพิ่ม
