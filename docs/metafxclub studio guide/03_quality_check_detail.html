Flow Step 2 · Quality Check

เช็คคุณภาพข้อมูลและ Feature ก่อน Train

บทนี้อธิบายขั้นก่อนเริ่ม train รอบจริง: ตรวจว่า CSV ใช้ได้, feature ไม่ซ้ำจนทำให้โมเดลจำ noise, และ regime ของตลาดเหมาะกับช่วงที่จะนำโมเดลไปใช้

ทำไมต้องเช็ค ขั้นตอนใช้งาน เช็คอะไรบ้าง Feature Analysis Regime Check ตัดสินใจไปต่อ

ทำไม Quality Check ต้องมาก่อน Train

Data

ข้อมูลผิด โมเดลก็ผิด

ถ้า timestamp เพี้ยน, rows น้อย, missing เยอะ หรือไฟล์ไม่มี .params.json ผล train/backtest จะตีความยากตั้งแต่ต้น

Feature

Feature ซ้ำทำให้ overfit ง่าย

Indicator หลายตัวอาจเล่าเรื่องเดียวกัน ถ้าใส่ซ้ำมากเกินไป agent อาจจำ noise และ neural network ใหญ่เกินจำเป็น

Regime

ตลาดคนละช่วงอาจคนละนิสัย

ข้อมูลย้อนหลังยาวมากอาจปนช่วง crisis, COVID, Brexit หรือ volatility shift ควรดู regime ก่อนตัดสินใจใช้ full history

จำง่าย ๆ: Quality Check คือด่านคัดข้อมูลก่อนให้โมเดลเรียน ไม่ใช่การทำให้ผลดูดีขึ้น แต่เป็นการลดโอกาสที่โมเดลเรียนจากข้อมูลผิดช่วง ผิดกติกา หรือ feature ที่ซ้ำกันมากเกินไป

ขั้นตอนใช้งานใน Dashboard

เวลาใช้สอน user ให้เริ่มจาก flow สั้น ๆ ตรงนี้ก่อน เพื่อให้เห็นว่าหน้า Quality Check ต้องเดินจากไฟล์ข้อมูลไปสู่ไฟล์ที่พร้อม train อย่างไร หลังจากนั้นค่อยลงรายละเอียดว่า Feature Analysis, Clean Redundant Features และ Regime Check แต่ละตัวทำหน้าที่อะไร

เช็คอะไรบ้างก่อน Train

หัวข้อ ควรเห็นอะไร ถ้าไม่ผ่านควรทำอะไร
ไฟล์และ sidecar มี .csv และถ้าจะ deploy ควรมี .params.json คู่กัน กลับไป import จาก DataCollector_RL ใหม่ หรือ attach params ให้ถูกไฟล์
เวลาและลำดับข้อมูล มี column timestamp, เรียงตามเวลาได้, ไม่มีช่วงย้อนกลับผิดปกติ แก้ไฟล์ต้นทางหรือ export ใหม่จาก MT5 ก่อน train
OHLCV และ feature หลัก มี open, high, low, close และ feature ไม่ว่างทั้ง column ตรวจ DataCollector_RL, indicator compile, symbol/timeframe และ warm-up bars
Feature correlation รู้ว่าคู่ feature ไหน correlate สูงกว่า threshold เช่น 0.85 ใช้ Clean Redundant Features หรือเลือก feature set ที่เล็กลง
Regime shift รู้ว่ามี breakpoint ใหญ่ไหม และควรใช้ full history หรือ cutoff dataset ใช้ Regime Check แล้วเลือก cutoff dataset ก่อนไปหน้า Train

Feature Analysis & Cleanup ใช้ทำอะไร

Show

Show Correlation Matrix

ใช้ดูว่า feature คู่ไหนสัมพันธ์กันสูง ระบบจะอ่าน numeric feature, ตัด column ที่ไม่ควรวิเคราะห์ เช่น timestamp, OHLCV, และชื่อที่เสี่ยง look-ahead เช่น future_, target

Output หลักคือรูป <csv_stem>_correlation.png และ log รายชื่อคู่ feature ที่เกิน threshold

Clean

Clean Redundant Features

ใช้ลบ feature ที่ซ้ำกันมากโดยดู correlation แบบ greedy: ถ้าคู่ใดเกิน threshold ระบบจะเก็บตัวที่สัมพันธ์กับ feature อื่นน้อยกว่า และ drop ตัวที่ซ้ำกว่า

Output คือไฟล์ใหม่ <csv_stem>_clean.csv โดยไฟล์เดิมยังอยู่ ไม่ถูกลบทิ้ง

วิธีอ่าน Correlation Matrix จากตัวอย่าง

ตัวอย่างจากไฟล์ uj_h4_dataset_correlation.png หลังจากกด Show Correlation Matrix ของ uj_h4_dataset.csv: รูปนี้มี 75 features และใช้ threshold 0.85

สิ่งที่เห็นในรูป แปลว่าอะไร ควรอธิบายกับ user อย่างไร
เส้นทแยงสีแดงเข้ม feature เทียบกับตัวเอง จึงได้ correlation = 1.00 ให้ ignore เส้นทแยงหลัก เพราะไม่ได้บอกปัญหา feature ซ้ำ
ช่องสีน้ำเงินเข้มจำนวนมาก feature คู่นั้นสัมพันธ์กันต่ำหรือแทบไม่ซ้ำกัน บริเวณนี้ไม่ใช่ปัญหาหลัก ไม่ต้อง clean เพราะสีเข้มอย่างเดียว
กลุ่มสีส้ม/แดงเป็นบล็อก feature หลายตัวในกลุ่มเดียวกันเล่าเรื่องใกล้กันมาก ให้มองเป็น candidate redundant cluster เช่นกลุ่ม trend, oscillator, volatility
ตัวเลข 0.85+ บนช่อง ระบบใส่ตัวเลขเฉพาะคู่ที่เกิน threshold ให้ดูตัวเลขและ log Top correlated pairs คู่กัน ไม่อ่านจากสีอย่างเดียว
บล็อกของ ema_20, ema_50, ema_100, ema_200, ema_long moving averages หลายตัวสัมพันธ์กันสูงมาก บางคู่ใกล้ 1.00 อธิบายว่าเป็น feature trend ที่ซ้ำกันบางส่วน การ clean อาจเก็บบางตัวและตัดบางตัวออก
กลุ่ม rsi_min, rsi_max, rsi_mean สถิติ RSI หลายแบบเคลื่อนตามกันสูง เช่นในรูปมีหลายช่องราว 0.90+ ให้ user เข้าใจว่า indicator family เดียวกันมักซ้ำกัน ไม่จำเป็นต้องใส่ทุก variant เสมอ
กลุ่ม atr_min, atr_max, atr_mean, atr_std volatility features ในตระกูล ATR อาจซ้ำกันสูง ถ้า clean แล้วลด feature กลุ่มนี้ ถือว่าเป็นการลด input ที่ซ้ำ ไม่ใช่ลบข้อมูลราคา
กลุ่ม candle pattern ส่วนใหญ่สีน้ำเงิน/กระจาย pattern บางตัวไม่ได้สัมพันธ์สูงกับ indicator กลุ่มหลัก ไม่ควรลบ candle features เพียงเพราะดูไม่เด่นในภาพ ต้องดู threshold/log เป็นหลัก
วิธีสอนแบบเร็ว: ให้ user ดูสามอย่างตามลำดับ: 1) มีบล็อกสีแดง/ส้มใหญ่ไหม, 2) log บอกคู่ไหนเกิน threshold บ้าง, 3) feature ที่ซ้ำเป็น family เดียวกันหรือไม่ ถ้าใช่ค่อยพิจารณา Clean Redundant Features

Clean Redundant Features ทำงานยังไง

ปุ่มนี้ไม่ได้ลบ feature แบบสุ่ม และไม่ได้แก้ไฟล์เดิมโดยตรง แต่จะสร้างไฟล์ใหม่ <csv_stem>_clean.csv ที่ตัด feature ซ้ำตาม correlation threshold

ขั้น โปรแกรมทำอะไร ความหมายสำหรับ user
1. เลือก feature ที่เอามาวิเคราะห์ ใช้เฉพาะ numeric feature และข้าม timestamp, symbol, OHLCV, volume, future_, target ระบบพยายามไม่เอา column ราคา/label/leaky column มาใช้ตัดสิน feature ซ้ำ
2. คำนวณ absolute correlation คำนวณค่า correlation แบบค่าสัมบูรณ์ระหว่าง feature ทุกคู่ ทั้งสัมพันธ์ทางเดียวกันและสวนทางกันแรง ๆ ถือว่าอาจซ้ำกันได้
3. หา feature pair ที่ซ้ำที่สุด ดูคู่ที่มี correlation สูงสุด และต้องสูงกว่า threshold เช่น 0.85 เริ่มแก้จากคู่ที่ซ้ำหนักที่สุดก่อน
4. เลือกว่าจะ drop ตัวไหน ในคู่ที่ซ้ำกัน ระบบคำนวณว่า feature ไหนมี average correlation กับ feature อื่นสูงกว่า แล้ว drop ตัวนั้น ตัวที่เชื่อมโยงกับคนอื่นเยอะกว่า มัก redundant กว่า จึงถูกตัดออก
5. ทำซ้ำจน clean วนซ้ำจนไม่เหลือคู่ feature ที่ correlation สูงกว่า threshold ผลลัพธ์คือชุด feature ที่เบาลงและซ้ำกันน้อยลง
6. บันทึกไฟล์ใหม่ เก็บ non-feature columns ไว้ แล้วต่อด้วย feature ที่ระบบเลือกเก็บ จากนั้น save เป็น _clean.csv ไฟล์เดิมยังอยู่ ให้ใช้ไฟล์ clean ไป train เปรียบเทียบได้
ข้อควรระวัง: ถ้าทำ strict train/test split แยกไฟล์ชัดเจน ควรเลือก feature จากฝั่ง train แล้วใช้ selection เดียวกันกับ test เพื่อเลี่ยงการใช้ข้อมูล test มาช่วยตัดสิน feature selection
ค่า / ปุ่ม ความหมาย คำแนะนำสำหรับสอน user
Correlation threshold ค่าที่ใช้บอกว่า feature สองตัวซ้ำกันมากแค่ไหน ค่า default คือ 0.85 เริ่มที่ 0.85 ก่อน ถ้าลบเยอะเกินให้เพิ่มเป็น 0.90
Show Correlation Matrix ดูภาพรวมก่อนว่ามีปัญหาไหม ใช้ก่อน clean เพื่อให้ user เห็นเหตุผล ไม่ใช่กดลบแบบไม่รู้ว่าเกิดอะไรขึ้น
Clean Redundant Features สร้าง CSV ใหม่ที่ตัด feature ซ้ำออก ใช้เมื่อ feature เยอะมากหรือ correlation สูงหลายคู่ แล้วค่อยนำไฟล์ _clean.csv ไป train

Regime Check ใช้ตอนไหน

Regime Check เป็นส่วนหนึ่งของ Quality Check เพราะมันช่วยตอบคำถามก่อน train ว่า เราควรใช้ข้อมูลทั้งประวัติ หรือควรเริ่ม train จากช่วงตลาดใหม่

ควรใช้

Dataset ย้อนหลังยาว

เช่นข้อมูลหลายปีหรือหลาย cycle ที่อาจผ่าน crisis, policy change หรือ volatility shift

ควรใช้

ผล train/backtest แปลก

เช่น train ดูดีแต่ test แย่ หรือช่วงหลังตลาดดูนิสัยไม่เหมือนอดีต

ระวัง

ไม่ใช่สัญญาณซื้อขาย

Regime Check ใช้เลือกช่วงข้อมูล ไม่ใช่ indicator สำหรับเปิด order โดยตรง

Method ใช้ดูอะไร เหมาะกับการสอนแบบไหน
HMM 3-state มองตลาดเป็น hidden states เช่น bull/bear/sideways จาก return และ volatility ใช้เป็นค่าแนะนำเริ่มต้น เพราะ dashboard ตั้งเป็น method หลัก
K-Means rolling จัดกลุ่ม rolling mean/std/slope เพื่อดูช่วงตลาดที่มีลักษณะคล้ายกัน เหมาะกับการ demo เพราะเร็วและเข้าใจง่าย
PELT หา changepoint ทางคณิตศาสตร์ โดยปรับ penalty เพื่อให้ตัดมาก/น้อย ใช้เมื่อต้องการมอง structural break แบบจริงจัง แต่รันช้ากว่า

ค่าแนะนำสำหรับ Regime Check

ค่า แนะนำให้ตั้ง เหตุผล / ข้อควรระวัง
Method HMM ใช้เป็นค่าเริ่มต้น เพราะเหมาะกับการหา hidden regime จากพฤติกรรมราคา โดยดู return และ volatility ไม่ได้ดูข่าวโดยตรง
Number of states 3 states ใช้ 3 ก่อน เพราะอธิบายง่ายและไม่แตก regime มากเกินไป: มองเป็นตลาดนิ่ง/ผันผวน/อีกสภาพตลาดหนึ่ง เช่น trend หรือ distribution ที่ต่างออกไป
Top-K shocks 15 ใช้สำหรับ refresh known events ด้วย Gemini เพื่อเอา event มาเทียบกับ breakpoint. ค่านี้ไม่ได้เปลี่ยนวิธีที่ HMM หา breakpoint โดยตรง
เปลี่ยนเป็น 2 states เมื่อไหร่ เฉพาะกรณีอยาก demo ง่ายมาก 2 states มักแปลเป็น low-vol/high-vol ได้ง่าย แต่หยาบเกินไปสำหรับการเลือกช่วง train จริง
ใช้ 4-5 states เมื่อไหร่ ใช้เฉพาะตอน advanced review state เยอะขึ้นอาจจับรายละเอียดได้มากขึ้น แต่เสี่ยง overfit, breakpoint เยอะเกิน และทำให้เลือก cutoff ยากขึ้น

เลือกช่วง Data จากผล Regime Check ยังไง

ขั้น ให้ดูอะไร ตัดสินใจยังไง
1. ดู breakpoint วันที่ที่ HMM บอกว่า hidden state เปลี่ยน และอยู่ต่อเนื่องพอสมควร ถือเป็น candidate date ที่ตลาดอาจเปลี่ยนพฤติกรรม ไม่ใช่คำสั่งให้ตัดข้อมูลทันที
2. ดู event match breakpoint ใกล้ event จริงไหม เช่น Brexit, COVID panic, CPI shock ถ้า match กับ event ใหญ่ในช่วงไม่กี่วัน/ไม่กี่สัปดาห์ น้ำหนักของ breakpoint นั้นสูงขึ้น
3. ดูจำนวนข้อมูลหลัง cutoff ถ้าเลือก breakpoint ล่าสุดมากเกินไป อาจเหลือข้อมูล train น้อย อย่าเลือก cutoff ที่ทำให้ dataset สั้นจน train/backtest ไม่น่าเชื่อถือ
4. สร้าง cutoff dataset เลือก breakpoint ที่สมเหตุสมผล แล้วกด Use Selected as Train Cutoff ระบบจะสร้างไฟล์ _from_YYYY-MM-DD.csv เพื่อนำไป train เทียบกับ full-history
5. เทียบผล train/backtest ทั้ง full-history และ cutoff dataset ถ้าจำเป็น เลือกชุดข้อมูลที่ทำให้ out-of-sample/backtest/walk-forward เสถียรกว่า ไม่ใช่เลือกจาก chart อย่างเดียว
หลักคิด: ถ้า Regime Check เจอ breakpoint ใหญ่และมีเหตุการณ์รองรับ เช่น crisis หรือ event สำคัญ ให้ลองสร้าง cutoff dataset ด้วย Use Selected as Train Cutoff แล้วนำไฟล์ใหม่ไป train/backtest เทียบกับ full-history

Decision: แบบไหนถึงไป Train ได้

สถานะ ความหมาย ตัดสินใจ
Go CSV มี rows พอ, timestamp ใช้ได้, feature ไม่เสีย, correlation ไม่หนัก, regime ไม่ขัดกับช่วงที่จะใช้งาน ไปหน้า Train ได้
Review feature ซ้ำเยอะ หรือ Regime Check เจอ breakpoint ใหญ่ ลอง _clean.csv หรือ cutoff dataset แล้ว train เปรียบเทียบ
Stop timestamp พัง, OHLCV ขาด, feature ว่างทั้ง column, ไฟล์ผิด format หรือข้อมูลน้อยเกิน กลับไป Step 1: Data Prep / DataCollector_RL ก่อน
ประโยคสอน user: อย่าเพิ่งถามว่า train กี่ steps จนกว่าจะตอบได้ว่า data ที่จะ train เป็นช่วงตลาดที่เหมาะ มี feature ที่ไม่ซ้ำเกินไป และไม่มีปัญหาไฟล์พื้นฐาน
