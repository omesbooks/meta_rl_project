Flow Step 5 · Robustness

Walk-Forward ตรวจว่า edge ทนหลายช่วงตลาดไหม

บทนี้อธิบายหน้า Walk-Forward Validation: ทำไมหน้านี้ถึง retrain ใหม่ทุก window, ทำไมมันอยู่หลัง Train/Backtest, วิธีอ่าน ROBUST/UNSTABLE/NOT ROBUST, และควรตัดสินใจอย่างไรเมื่อผลออกมาไม่ผ่าน

เป้าหมาย ไม่ใช่ทดสอบ model file ทำไมอยู่หลัง Train วิธีแบ่ง Window ค่าบนหน้า Dashboard ขั้นตอนใช้งาน อ่านผล ตัดสินใจ Confidence Filter

Walk-Forward คืออะไรใน flow นี้

Robustness

ตรวจหลายช่วงเวลา

แทนที่จะดูแค่ backtest รอบเดียว ระบบจะหั่นข้อมูลเป็นหลาย window แล้วดูว่าผลยังพอผ่านในหลายช่วงตลาดหรือไม่

OOS

Train แล้ว test ช่วงถัดไป

แต่ละ window จะ train ด้วยช่วงอดีต แล้ว test กับข้อมูลถัดไปที่ model ยังไม่เคยเห็น เพื่อลดการหลอกตัวเองจากผล in-sample

Gate

ใช้ก่อน Export

ถ้า Walk-Forward fail หลาย window แปลว่า edge ยังไม่น่าไว้ใจ แม้ backtest รอบเดียวอาจดูดี

จำง่าย ๆ: Backtest ตอบว่า model รอบนี้พอมีหวังไหม ส่วน Walk-Forward ตอบว่า วิธี train ชุดนี้ยังพอทำงานได้ไหมเมื่อเลื่อนเวลาไปหลายช่วง

จุดที่ต้องเข้าใจก่อน: หน้านี้ไม่ได้ทดสอบ model file ที่ train ไว้แล้ว

หน้า Walk-Forward ของโปรแกรมนี้ไม่มีช่องเลือก model เพราะมันไม่ได้เอาไฟล์ .zip ที่ train จากหน้า Train มา validate โดยตรง แต่จะใช้ dataset และค่าตั้งต้น แล้วสร้าง model ชั่วคราวใหม่ในแต่ละ window

เข้าใจผิดง่าย

ไม่ใช่ saved model validation

ถ้าต้องการทดสอบ model ที่ train ไว้แล้วจริง ๆ ให้ใช้หน้า Backtest โดยเลือก model, CSV และช่วงทดสอบ

ความหมายจริง

เป็น training recipe validation

Walk-Forward ใช้พิสูจน์ว่า recipe นี้น่าเชื่อถือไหม: dataset, feature set, window size, steps per window, reward logic และวิธี train

Walk-Forward ในโปรแกรมนี้ = validate training recipe / strategy process, not the saved model file
ประโยคสอน user: ทำหลัง Train ไม่ได้แปลว่าเอา model file ที่ train แล้วไปใช้ต่อ แต่แปลว่าเรามี setup ที่ดูมีหวังแล้ว จึงนำ setup นั้นไปทดสอบซ้ำหลายช่วงเวลา

ทำไม Walk-Forward ถึงอยู่หลัง Train/Backtest

ลำดับ หน้าที่ เหตุผล
Train หา candidate setup ใช้เลือกชุดข้อมูล, reward mode, window size, steps, max hold และ hyperparameter ตั้งต้น
Backtest คัดกรองเร็ว ถ้า backtest รอบเดียวพังหนัก ไม่คุ้มรัน Walk-Forward เพราะเสียเวลาหลายรอบ
Walk-Forward พิสูจน์ robustness ถ้า setup ที่ดูดีในรอบเดียว ยังพอผ่านหลาย window แปลว่า edge มีน้ำหนักมากขึ้น
หลักคิด: Walk-Forward เป็นด่านหนักและใช้เวลานาน จึงควรทำหลังจากมี setup ที่พอมีหวังแล้ว ไม่ใช่ใช้เป็นเครื่องมือทดลองสุ่มทุก dataset ตั้งแต่ต้น

ระบบแบ่ง Window ยังไง

ค่าเริ่มต้นคือ 5 windows และใช้ mode แบบ rolling. ในแต่ละ window ระบบจะ normalize ด้วยข้อมูล train ของ window นั้น แล้วนำ mean/std เดียวกันไปใช้กับ test slice เพื่อเลี่ยง look-ahead

Window Train Test OOS ความหมาย
1 0%-50% 50%-60% train ครึ่งแรก แล้ว test ช่วงถัดไป
2 10%-60% 60%-70% เลื่อนทั้ง train และ test ไปข้างหน้า
3 20%-70% 70%-80% ทดสอบว่าช่วงกลางถึงปลายยังไหวไหม
4 30%-80% 80%-90% ทดสอบช่วงใหม่กว่าเดิม
5 40%-90% 90%-100% ทดสอบช่วงท้ายสุด คล้ายตลาดล่าสุดที่สุด
หมายเหตุ: CLI รองรับ anchored ด้วย แต่หน้า Dashboard ปัจจุบันใช้ค่าเริ่มต้นแบบ rolling คือ train window เลื่อนไปข้างหน้าพร้อมกับ test window

ค่าหลักในหน้า Walk-Forward

ช่อง ทำหน้าที่อะไร คำแนะนำ
Dataset CSV ที่จะใช้ retrain/test หลาย window ควรเป็นไฟล์ที่ผ่าน Quality Check และ Regime Check แล้ว
Windows จำนวนช่วงที่จะรัน walk-forward ค่าเริ่มต้น 5 ยิ่งเยอะยิ่งละเอียด แต่ใช้เวลานานขึ้นและ test slice อาจบางลง
Steps/window จำนวน training steps ต่อ window ค่าเริ่มต้น 50000 ควรสัมพันธ์กับขนาดข้อมูล ถ้าน้อยเกิน model อาจยังเรียนไม่ทัน
Window size จำนวน bars ย้อนหลังที่ agent เห็นเป็น state ควรสอดคล้องกับ setup ที่ใช้ในหน้า Train เพื่อทดสอบ recipe เดียวกัน
Output Name prefix สำหรับไฟล์ผลลัพธ์ เช่น wf_prod จะได้ wf_prod_results.csv และ wf_prod_chart.png

ขั้นตอนใช้งานใน Dashboard

1
Prepare

เลือก dataset ที่ผ่านขั้นก่อนหน้าแล้ว

ใช้ไฟล์ที่ผ่าน Data Prep, Quality Check และ Regime Check แล้ว ถ้าเพิ่งสร้าง cutoff dataset ให้เลือกไฟล์ _from_YYYY-MM-DD.csv

2
Match Recipe

ตั้งค่าให้ใกล้กับ setup ที่จะใช้จริง

ตั้ง Windows, Steps/window, Window size ให้สะท้อนสูตร train ที่ต้องการพิสูจน์ ไม่ใช่ตั้งแบบสุ่ม

3
Run

กด Start Walk-Forward Retrain

ระบบจะ train model ชั่วคราวใหม่ในแต่ละ window แล้ว test OOS ช่วงถัดไป กระบวนการนี้อาจใช้เวลานานกว่าการ backtest ปกติมาก

4
Read

อ่าน verdict และผลราย window

ดูว่ากี่ window ผ่าน, PF เฉลี่ยเท่าไร, return เฉลี่ยเป็นบวกไหม, drawdown แย่สุดรับได้หรือไม่ และมี window ไหนพังหนักเป็นพิเศษ

5
Decide

ตัดสินใจว่าจะ export หรือย้อนกลับ

ถ้า robust ให้กลับไปยืนยัน final model/backtest ก่อน export. ถ้า not robust ให้ย้อนกลับไป Regime Check, feature, steps หรือ training setup

อ่านผล Walk-Forward อย่างไร

Metric / Verdict ความหมาย อ่านแบบเร็ว
WR Win rate ของแต่ละ window โค้ดใช้ WR > 50% เป็นส่วนหนึ่งของเกณฑ์ ROBUST เต็มรูปแบบ
PF Profit Factor: กำไรรวม / ขาดทุนรวม PF > 1 แปลว่าช่วงนั้นกำไรมากกว่าขาดทุน
Return ผลตอบแทนของ test slice นั้น ควรดูทั้งค่าเฉลี่ยและ window ที่แย่สุด ไม่ใช่ดูแค่ max
DD Max drawdown ของแต่ละ window ถ้า worst DD หนักเกินรับได้ ต่อให้ PF เฉลี่ยดี ก็ยังต้อง review
ROBUST ทุก window ผ่านเกณฑ์ PF > 1 และ WR > 50% ไปขั้น final confirmation/backtest/MT5 tester ได้
MOSTLY ROBUST window ส่วนใหญ่มี PF > 1 แต่ยังไม่ครบทุกเงื่อนไข ใช้ระวัง ต้องดู window ที่ fail ว่าเกิดจาก regime หรือ sample น้อย
UNSTABLE ประมาณครึ่งหนึ่งของ window ผ่าน ยังไม่ควร deploy ให้กลับไปปรับ data/recipe แล้วรันใหม่
NOT ROBUST window ส่วนใหญ่ fail ถือว่ายังไม่ผ่าน ห้าม export ไปใช้งานจริงจากผลชุดนี้

Decision: ถ้าผลออกมาแบบต่าง ๆ ต้องทำอะไร

ผลลัพธ์ ตีความ ทางไปต่อ
ROBUST recipe มีหลักฐานว่าทนได้หลายช่วงเวลา ยืนยัน final model ด้วย Backtest/MT5 Strategy Tester แล้วค่อย Export/Deploy
MOSTLY ROBUST พอมี edge แต่ยังมีช่วงที่ fail ดูว่า fail window อยู่ช่วง event/regime ไหน แล้วอาจแยก dataset หรือปรับ risk
UNSTABLE ผลยังแกว่งมาก อาจ overfit หรือ data ปนหลาย regime กลับไป Regime Check, Feature Analysis, steps/window หรือ reward setup
NOT ROBUST edge ไม่อยู่หลายช่วง ผลอาจขึ้นกับ luck หยุด flow deploy และเริ่มปรับ data/train recipe ใหม่
ข้อห้ามสำคัญ: ถ้า Walk-Forward เป็น NOT ROBUST อย่าแก้ด้วยการ export ไปลองจริง เพราะปัญหาคือ recipe ยังไม่ทนหลายช่วงเวลา ไม่ใช่แค่ยังไม่ได้ deploy

Walk-Forward ใช้ Confidence Filter ไหม

Walk-Forward ในโปรแกรมนี้ train temporary model แล้วประเมินด้วย environment แบบใกล้กับตอน train: โมเดลเลือก action ด้วย policy โดยตรง และไม่ได้ใช้ confidence threshold แบบหน้า Backtest/Live. ดังนั้นผล Walk-Forward ตอบว่า training recipe มี edge หลายช่วงเวลาไหม แต่ยังไม่ได้ตอบว่า threshold 0.85 หรือ threshold อื่นเหมาะกับการส่ง order จริงหรือไม่

What it proves

recipe มี edge แบบ pure policy ไหม

ถ้าหลาย window ผ่าน แปลว่า dataset + feature + reward + window size + steps มีน้ำหนักมากขึ้นว่าไม่ได้ดีเฉพาะช่วงเดียว

What it does not prove

ยังไม่ calibrate live filter

ถ้าจะใช้ confidence filter ตอน live ต้องกลับไปหน้า Backtest/Analyze แล้ว sweep threshold บน OOS ไม่ควรเดาว่า 0.85 ดีเสมอ

ลำดับที่แนะนำ: Train/Walk-Forward ใช้ดู pure policy edge ก่อน จากนั้น Backtest แบบ Confidence = 0 เพื่อเทียบกับ train และค่อย sweep confidence threshold สำหรับ production gate

คำถามที่ user มักงง

คำถาม คำตอบสั้น อธิบายเพิ่ม
ทำไมไม่มีช่องเลือก model? เพราะหน้านี้ไม่ได้ test model เดิม มัน train temporary model ใหม่ในแต่ละ window เพื่อ validate recipe
แล้ว model ที่ train ไว้ใช้ทดสอบที่ไหน? ใช้หน้า Backtest Backtest คือหน้าที่เลือก saved model แล้วทดสอบกับ CSV/ช่วงข้อมูลที่ต้องการ
ถ้า Walk-Forward ผ่าน ต้อง train ใหม่ไหม? ถ้าเปลี่ยน recipe ให้ train final model ใหม่ โมเดลที่จะ deploy ควรสร้างจาก recipe เดียวกับที่ผ่าน Walk-Forward และต้อง backtest ยืนยันอีกครั้ง
ถ้า fail ควรกลับไปจุดไหน? กลับไป Quality/Regime/Train ดูว่า fail เพราะ data ปน regime, feature ซ้ำ, steps น้อย หรือ setup ไม่เหมาะกับตลาดล่าสุด
Walk-Forward ผ่านแล้วใช้ confidence 0.85 ได้เลยไหม? ยังไม่ได้ Walk-Forward ไม่ได้ calibrate confidence filter ต้องใช้ Backtest/Analyze sweep threshold เพิ่มก่อน deploy
