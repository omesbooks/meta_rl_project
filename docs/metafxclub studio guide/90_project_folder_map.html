Appendix · Project Map

แผนที่โฟลเดอร์ของโปรเจกต์ Meta RL Trading

บทนี้อธิบายโฟลเดอร์ที่เห็นใน root ของโปรเจกต์ ว่าแต่ละที่เก็บอะไร ใช้ตอนไหน และโฟลเดอร์ไหนควรปล่อยให้ระบบจัดการเอง เพื่อไม่ให้ไฟล์โมเดล, backtest, MT5 package, docs และ source code ปนกันจนติดตั้งผิดหรือ debug ยาก

กฎจำง่าย ตารางโฟลเดอร์ ไฟล์ที่ระบบสร้าง ไฟล์ที่ยังอยู่ root หาไฟล์จากตรงไหน ลบ/ย้ายอะไรได้บ้าง

กฎจำง่ายก่อนดูรายละเอียด

Code

root เก็บ source หลัก

ไฟล์อย่าง rl_app.py, rl_train.py, backtest_live.py, export_to_onnx.py คือแกนของระบบ RL ใช้แก้ logic ของโปรแกรม ไม่ใช่ที่เก็บผลลัพธ์โมเดลระยะยาว

Artifacts

artifacts/ คือบ้านของโมเดล

หลัง train/backtest ให้มองหาไฟล์โมเดล, norm, logs, best model, trades CSV, equity chart และ metadata ใน artifacts/models/<model>/

MT5

mt5_files/ คือบ้านของ MT5

template และ helper MQL5 อยู่ใน mt5_files/MQL5/ ส่วน package ที่ export เป็นชุดติดตั้งต่อโมเดลอยู่ใน mt5_files/packages/<deploy_name>/MQL5/

จำสั้น ๆ: เขียนโค้ดที่ root, อ่านคู่มือใน docs, ดูผล train/backtest ใน artifacts, เอาไฟล์ไป MT5 จาก mt5_files/packages.

โฟลเดอร์ใน root ใช้ทำอะไร

โฟลเดอร์ หน้าที่ ควรแตะไหม ตัวอย่าง/ข้อควรระวัง
.agents/ พื้นที่ local ของ agent/tooling บางตัว ใช้เก็บ config หรือ state เฉพาะเครื่อง ไม่ควรแก้มือ ในโปรเจกต์นี้ตอนตรวจมีโฟลเดอร์อยู่ แต่ไม่มีไฟล์ข้างใน ปล่อยไว้ได้
.codex/ พื้นที่ local สำหรับ Codex/session/tool metadata ใน workspace นี้ ไม่ควรแก้มือ ใช้โดยเครื่องมือ ไม่ใช่ส่วน runtime ของ RL app
.git/ ฐานข้อมูล Git: history, branches, refs, hooks และ config ของ repo ห้ามแก้ตรง ๆ ถ้าพัง repo อาจเสีย ใช้คำสั่ง Git เท่านั้น
.venv/ Python virtual environment ของโปรเจกต์ เก็บ interpreter และ packages ที่ติดตั้ง ลบได้เมื่อจะสร้างใหม่ รันโปรเจกต์ควรใช้ .venv/Scripts/python.exe ไม่ใช้ Python ตัวอื่น
__pycache__/ Python bytecode cache ที่สร้างอัตโนมัติจากการ import ไฟล์ .py ลบได้ ถ้าจะลบ ปิดโปรแกรมก่อน ระบบจะสร้างใหม่เองเมื่อรัน Python
artifacts/ พื้นที่รวม output ที่ระบบสร้างหลัง train/backtest/chart ต่อโมเดล ดูผลที่นี่ ใช้โครงสร้าง artifacts/models/<model>/ เพื่อไม่ให้ไฟล์โมเดลไปกองใน root
assets/ ไฟล์ภาพและทรัพยากรประกอบ UI เช่น logo/branding ของ GUI แก้ได้ถ้าปรับ UI ตัวอย่างตอนตรวจ: metafx.png, README.md
docs/ เอกสาร HTML/Markdown ของโปรเจกต์ รวมคู่มือ dashboard และ explainer ต่าง ๆ เขียนคู่มือที่นี่ ชุดคู่มือหลักอยู่ที่ docs/metafxclub studio guide/
examples/ ตัวอย่างการใช้งานหรือสคริปต์ตัวอย่างสำหรับอ้างอิง อ่าน/ทดลองได้ ตัวอย่างตอนตรวจ: rl_example.py ไม่ใช่ flow production หลัก
legacy/ โค้ดหรือไฟล์แนวทางเก่าที่เก็บไว้เทียบย้อนหลัง เช่น PyCaret/MQL รุ่นเก่า อ่านเพื่ออ้างอิง งาน RL ปัจจุบันไม่ควรเริ่มจาก folder นี้ ยกเว้นตั้งใจเทียบกับระบบเก่า
mt5_files/ พื้นที่สำหรับ MQL5, template, indicators, include และ package ที่ export ไปติดตั้งใน MT5 ใช้ตอน deploy/tester template อยู่ใน mt5_files/MQL5/; output ต่อโมเดลอยู่ใน mt5_files/packages/
reference/ คลังความรู้และเอกสาร research เช่น ML4T, quant reference, theory ที่ใช้ประกอบการออกแบบ อ่านประกอบ ไม่ใช่ runtime ของ app แต่ช่วยอธิบายหลักการ RL/backtest/feature engineering
tools/ สคริปต์ช่วยงาน maintenance, analysis, automation, data และ MT5 utilities ใช้เมื่อรู้เป้าหมาย ไม่ใช่ปุ่มหลักใน GUI ทุกตัว บางสคริปต์ไว้ช่วย debug หรือจัดระเบียบไฟล์

ไฟล์อะไรที่ยังอยู่ root ได้

ประเภทไฟล์ ควรอยู่ตรงไหน เหตุผล
Source script หลัก root เช่น rl_app.py, rl_train.py, backtest_live.py, backtest_chart.py, export_to_onnx.py เป็น entry point ของระบบ
Dataset CSV สำหรับ train/test root ตอนนี้ยังใช้ได้ GUI และ CLI เดิมค้นไฟล์ CSV จาก root เป็นหลัก ดังนั้นไฟล์อย่าง uj_h4_dataset.csv และ uj_h4_dataset.params.json ยังอยู่ root ได้ เพื่อให้เลือกจาก dropdown ได้ง่าย
Model/backtest output ใหม่ artifacts/models/<model>/ ไม่ควรให้ .zip, norm, logs, trade CSV และ chart ไปกอง root เพราะแยกไม่ออกว่าเป็นของโมเดลไหน
MT5 deployment output ใหม่ mt5_files/packages/<deploy>/MQL5/ ทำให้ copy ไป MT5 ได้เป็นชุดเดียว และลดความเสี่ยงหยิบ EA/config/ONNX ผิดโมเดล

ถ้าต้องหาไฟล์ ให้เริ่มจากตรงไหน

ของระบบ

.git/, .venv/, __pycache__/

ไม่ใช่ที่หา output โมเดล ถ้าไม่ได้แก้ environment หรือ Git ไม่ต้องเปิดดู

ลบหรือย้ายอะไรได้บ้าง

รายการ คำแนะนำ เหตุผล
__pycache__/ ลบได้ เป็น cache อัตโนมัติ ปิดโปรแกรมก่อนแล้วลบได้ ระบบสร้างใหม่เอง
.venv/ ลบได้เมื่อจะ rebuild ลบแล้วต้องติดตั้ง requirements ใหม่ ไม่ควรลบระหว่างกำลังใช้งาน GUI
artifacts/models/<model>/ ลบได้ถ้าไม่ใช้โมเดลนั้นแล้ว ลบแล้วผล train/backtest ของโมเดลนั้นจะหาย ควร backup ถ้ายังต้องเทียบผล
mt5_files/packages/<deploy>/ ลบได้ถ้า package เก่า เป็น output จาก export สร้างใหม่ได้ แต่ถ้าติดตั้งใน MT5 อยู่ให้เช็คก่อน
.git/ ห้ามลบ คือประวัติ repo ทั้งหมด ถ้าลบจะไม่ใช่ Git repo เดิม
mt5_files/MQL5/Experts/*_template.mq5 ไม่แก้เฉพาะโมเดล เป็น template กลางสำหรับ export ถ้าจะเปลี่ยน behavior ทุกโมเดลค่อยแก้ด้วยความตั้งใจ

แนวคิดหลักคือ “ลบ cache ได้, ลบ output ได้เมื่อไม่ใช้แล้ว, อย่าลบ history/tooling/source/templates โดยไม่ตั้งใจ” เพื่อให้โปรเจกต์ยัง regenerate ผลลัพธ์เดิมได้จาก source code และ dataset
