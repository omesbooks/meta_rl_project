🧠 Stage 3: Create PPO Agent

3 PPO Neural Network

Proximal Policy Optimization — algorithm ที่ stable + sample-efficient
มี 2 networks: Actor (เลือก action) และ Critic (ประเมิน state)

📋 PPO Setup

from stable_baselines3 import PPO model = PPO( "MlpPolicy", env, learning_rate=3e-4, n_steps=2048, # trajectory length batch_size=64, n_epochs=10, # SGD passes gamma=0.99, # discount gae_lambda=0.95, clip_range=0.2, # PPO core idea ent_coef=0.01, # exploration policy_kwargs={"net_arch": [128, 64]} )

🏗️ NN Architecture

Input layer: 353 nodes
  ↓
Hidden 1: 128 nodes (ReLU)
  ↓
Hidden 2: 64 nodes (ReLU)
  ↓ ↓
Actor head: 4 outputs (logits)
  → softmax → action probs
Critic head: 1 output
  → V(s) value estimate

Total params: ~50,000

🔁 Stage 4: Training Loop (หัวใจของ RL)

4 ทุก step ที่เกิดขึ้น (เก็บ + เรียน)

PPO ทำซ้ำ 200,000 steps. ทุก 2,048 steps จะ update weights ของ NN ครั้งหนึ่ง

🎬 ลำดับเหตุการณ์ในแต่ละ step

1. Observe
Env ส่ง state s_t ให้ agent (window of features + position info)
2. Forward
Agent NN: state → action probabilities [0.10, 0.65, 0.20, 0.05]
Hold 10%, Buy 65%, Sell 20%, Close 5%
3. Sample
สุ่มเลือก action ตาม prob (เพื่อ explore) — เช่น เลือก Buy (1)
4. Execute
Env รับ action → ทำให้เกิดผล (เปิด long position) → คำนวณ reward
5. Store
เก็บ tuple (s_t, a_t, r_t, s_t+1) ใน rollout buffer
6. Repeat
ทำซ้ำขั้น 1-5 จนกว่าจะครบ 2,048 steps
7. Update
⭐ PPO Update — ปรับ NN weights:
a) คำนวณ Advantage (action นี้ดีกว่าค่าเฉลี่ยแค่ไหน)
b) Gradient ascent บน clipped objective
c) ทำ 10 epochs ของ mini-batch SGD
→ Policy เก่งขึ้นเล็กน้อย
8. Repeat all
กลับไปขั้น 1 → ทำใหม่ → จนครบ 200,000 steps (~98 updates)

★ PPO Update — เกิดอะไรขึ้นทาง math

เมื่อเก็บ 2,048 transitions แล้ว PPO ปรับ NN ตามนี้:

📐 Advantage Estimation

# GAE (Generalized Advantage Estimation) # สำหรับแต่ละ step t: # 1. Discounted return G_t = r_t + γ·r_(t+1) + γ²·r_(t+2) + ... γ = 0.99 # discount factor # 2. Critic ทาย expected value V_t = critic_network(s_t) # 3. Advantage A_t = G_t - V_t # A_t > 0 → action ดีกว่าคาด → push prob up # A_t < 0 → action แย่กว่าคาด → push prob down

✂️ Clipped Objective

# Compare new vs old policy ratio = π_new(a|s) / π_old(a|s) # Clipped surrogate loss loss = -min( ratio * A, clip(ratio, 0.8, 1.2) * A ) # clip_range=0.2 → ห้าม policy เปลี่ยนเร็วเกิน # นี่คือ PPO core idea — stable! # Optimize optimizer.zero_grad() loss.backward() optimizer.step()

⚠️ Reward Shaping — บทเรียนสำคัญ

วิธีออกแบบ reward ส่งผลต่อพฤติกรรม agent มหาศาล

❌ MTM Mode (Buy-and-Hold Trap)

reward = (equity_t - equity_t-1) / equity_t-1
"ทุก bar ที่ราคาขึ้น → +reward → agent ถือยาว ไม่ปิด"
ผล: agent เปิด long ครั้งเดียว ถือ 3 ปี! 🐝

✅ Realized Mode (Smart)

reward = 0 (ขณะถือ) reward = pnl × 10 (ตอนปิด) + force close หลัง 30 bars
"ต้องปิด trade ถึงได้ reward → agent เรียนเทรดจริง"
ผล: 1,309 trades, WR 52%, PF 1.02 ✅

⚙️ Hyperparameters Explained

Parameter Value Purpose Impact
learning_rate 3e-4 ความเร็วในการ update weights ↑ เร็วเกิน → unstable, ↓ ช้าเกิน → ไม่เรียน
n_steps 2048 จำนวน steps ก่อน update ↑ trajectory ยาว → advantage แม่น
batch_size 64 Mini-batch ใน SGD standard
n_epochs 10 วน training data กี่รอบต่อ update ↑ เรียนแม่น แต่อาจ overfit
gamma (γ) 0.99 Discount factor 0.99 = มอง ~100 steps อนาคต
gae_lambda 0.95 Bias-variance ใน GAE standard
clip_range 0.2 จำกัด policy update ⭐ PPO core idea — ห้ามเปลี่ยนเร็ว
ent_coef 0.01 Entropy bonus ↑ explore มาก, ↓ exploit มาก
net_arch [128, 64] Hidden layer sizes ↑ ใหญ่ = capacity มาก แต่ overfit ง่าย
window_size 10 Bars history ที่ agent เห็น 10 = sweet spot, 100 = overfit แน่
max_hold_bars 30 บังคับปิดหลัง N bars กัน buy-and-hold trap

⚠️ Common Pitfalls ที่เจอ

🪤 Buy-and-Hold Trap

อาการ: Total trades = 1, WR = 100%, agent เปิด long ครั้งเดียวถือยาว
✅ แก้: realized reward + max_hold_bars

📊 Class Imbalance (Lazy Classifier)

อาการ: FLAT 73% → agent ทาย Hold ทุกครั้ง = accuracy 73%
✅ แก้: relabel.py quantile 33/33/33

🔮 Data Leakage

อาการ: WR 100% หลอกๆ — มี future_return ใน features
✅ แก้: Drop leaky cols ก่อน train

📈 Overfitting (Window Too Big)

อาการ: window=100 → 3,500 inputs → train ดี test แย่
✅ แก้: ใช้ window 10-30, sample/feature ratio > 50

🌊 Concept Drift

อาการ: Walk-forward window 4-5 (ปีล่าสุด) PF ตก 0.86
✅ แก้: Periodic retrain ทุก 3-6 เดือน

🎲 Reward Hacking

อาการ: Agent หาช่องโหว่ในการได้ reward (เช่น เปิด-ปิด ถี่ๆ)
✅ แก้: เพิ่ม cost penalty ให้แรงพอ
